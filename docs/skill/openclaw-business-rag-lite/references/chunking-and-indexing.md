# Chunking And Indexing

## 1. 切块原则

优先按 Markdown 标题切块，而不是固定字数硬切。

推荐顺序：
1. 按 `# / ## / ###` 分段
2. 每段过长时，再做二次切分
3. 保留标题路径到元数据中

## 2. 推荐 chunk 大小

对这套中文业务文档，建议：
- 目标块：`400-1200` 汉字
- 过短块：与相邻同层标题合并
- 过长块：在表格或列表边界拆分

不要这样切：
- 把一张表拆成很多碎块
- 把一个状态机说明和它的限制条件拆开
- 把“风险点”从模块说明中切散

## 3. 特殊文档切块建议

### 总表类文档
例如：
- `state-matrix.md`
- `business-metrics.md`
- `cross-module-writeback.md`

建议按二级标题整段切，必要时表格一张表一个 chunk。

### 模块正文
建议按这些区块优先切：
- 模块定位
- 主流程
- 状态流
- 联动对象
- 风险点

### 图谱类文档
建议按图说明整体成块，不要过度切碎。

## 4. 轻量向量库实现建议

优先级建议：

### 方案 A：SQLite 单文件

适合：
- 本地仓库内使用
- 调试方便
- 希望可直接查看元数据

建议表结构最少包含：
- `chunks`
- `embeddings`
- `manifests`

### 方案 B：JSONL + 本地向量文件

适合：
- 依赖最少
- 快速原型
- 先验证 skill 工作流

最小结构：

```text
metadata.jsonl
vectors.npy
manifest.json
```

### 方案 C：SQLite 元数据 + 向量外存

适合：
- 语料继续增长
- 需要更方便地调试过滤条件

## 5. 更新策略

建议采用“全量重建优先”的轻量策略：
- 文档变化不频繁时，直接全量重建索引
- 不要一开始就做复杂增量更新

只有当语料规模继续明显增长时，再考虑：
- 按文件 hash 增量更新
- 按目录分批重建
