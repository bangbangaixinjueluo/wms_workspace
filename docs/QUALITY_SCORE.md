# QUALITY_SCORE

## 说明
以下评分基于本次对 `wms-web`、`wms-api` 和现有文档的静态分析，不代表完整线上质量结论，但可以作为当前代码健康度的快速基线。

## 当前评分
### 架构清晰度：4/5
- 前端按业务域拆分较清楚，`views/wh` 与 `api/wh` 对应关系明显。
- 后端采用 Maven 多模块，模块边界总体清晰。
- 但随着私有业务继续叠加，模块规模已经开始变大。

### 业务承载能力：4/5
- 项目覆盖系统管理、工作流、报表、仓储和多种企业业务能力。
- 已有较成熟的管理后台基础能力和复杂表格支撑。

### 可维护性：3/5
- 前端 `main.js` 过重，全局注入较多。
- 后端配置、脚本和模块数量较多，对新人不够友好。
- 文档此前和代码脱节，本次已做第一轮回填，但仍需继续细化。

### 自动化保障：2/5
- 后端构建默认跳过测试。
- 前端未看到成体系的自动化测试沉淀。
- 回归验证大概率依赖人工。

### 环境治理：2/5
- 前后端存在较强环境耦合。
- 明文配置和真实资源地址直接出现在代码仓库，是明显短板。

### 安全基线：2/5
- 权限与认证基础能力存在。
- 但明文凭据、调试开关和本地配置问题让整体安全得分被明显拉低。

### 交付文档：3/5
- 目录已经有框架，但内容多数是模板。
- 目前已完成一轮贴合代码的修正，后续需要继续下钻到业务域和部署域。

## 综合判断
- 当前综合质量建议评为 `3/5`。
- 这是一个“业务能力较强、架构基本清晰，但工程治理仍需补课”的项目。

## 最值得优先投入的改进方向
1. 清理并迁移敏感配置，修复环境治理问题。
2. 建立最小自动化回归链路。
3. 为 WMS 核心业务补齐状态流转与接口文档。
4. 对前端初始化和后端配置做分层收敛，降低维护门槛。
