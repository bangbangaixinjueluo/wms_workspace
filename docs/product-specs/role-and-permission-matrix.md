# 角色与权限矩阵

## 目的
本文基于当前前端权限实现和接口命名，总结项目中的权限工作方式，帮助排查“为什么用户看不到菜单 / 按钮 / 数据 / 操作入口”。

## 当前权限实现结构
项目至少存在四层权限控制：

### 1. 登录态权限
- 通过 token 识别用户会话
- 无 token 时统一跳转登录页

### 2. 菜单/路由权限
- 前端调用 `/system/auth/list-menus`
- 后端返回当前用户可见菜单树
- 前端在 `store/modules/permission.js` 中转换为实际路由

### 3. 按钮/操作权限
- 用户信息接口 `/system/auth/get-permission-info` 返回 `permissions`
- 前端通过 `checkPermi` / `checkPermiAnd` 等方式控制按钮显示
- 首页快捷入口也会按权限码动态追加

### 4. 后端接口权限
- 即使前端展示了按钮，最终仍需后端接口权限兜底
- 关键写操作应默认以后端校验为准

## 当前可见权限对象
### 菜单
- 来源：后端菜单树
- 影响：左侧菜单、顶部菜单、可访问页面

### 角色
- 来源：`get-permission-info`
- 影响：角色判断、极少数角色级分支逻辑

### 权限码
- 来源：`get-permission-info`
- 影响：按钮、快捷入口、局部页面动作

### 数据范围
- 后端文档和框架中存在数据权限能力
- 这通常影响“能不能看见某些单据、客户、部门、仓库数据”

## 当前已识别的权限场景
### 系统级
- 用户、角色、菜单、字典、配置、任务、日志

### WMS 基础资料
- 仓库、库位、网点、主数据、客户、供应商、业务员

### 单据类
- 新建
- 审核
- 反审核
- 作废
- 接收
- 打印
- 导出

### 报表类
- 查询
- 导出
- 查看明细

## 典型问题定位方式
### 看不到菜单
- 检查 `/system/auth/list-menus` 返回结果
- 检查前端路由转换是否成功

### 进得去页面但按钮不显示
- 检查 `/system/auth/get-permission-info`
- 检查按钮绑定的权限码是否正确

### 有按钮但接口报无权限
- 说明前端与后端权限码可能不一致
- 也可能是后端存在更细的数据权限或接口权限控制

### 只能看到部分数据
- 优先怀疑部门、用户、租户或业务范围数据权限

## 验证建议
- 用至少两类角色账号做对比联调
- 同时验证菜单、按钮、接口、数据范围四层权限
- 对库存、财务、作废、反审核类动作单独复核
