# API_CONVENTIONS

## 目的
本文总结当前前后端代码中已经体现出来的接口交互约定，供后续开发和联调参考。

## 基础前缀
- 前端默认请求前缀：`${VUE_APP_BASE_API}/admin-api/`
- 开发代理通常代理 `/admin-api`

## 认证方式
- 使用 `Authorization: Bearer <token>`
- 前端统一在 `src/utils/request.js` 中注入 token
- 当开启租户模式时，请求头可附带 `tenant-id`

## 前端附加请求头
当前前端会额外附带以下页面上下文，便于后端日志记录和问题定位：

- `Page-Full-Path`
- `Page-Title`
- `Page-Name`
- `Page-Path`

## 返回结构
从前端响应处理逻辑可以看出，项目当前默认依赖以下字段：

- `code`
- `msg`
- `data`

其中：
- `code === 200` 视为成功
- `code === 401` 触发登录失效或刷新 token 逻辑
- `code === 500` 显示后端错误信息
- 其他业务码按项目错误码与提示策略处理

## Token 刷新
- 当前前端内置 refresh token 刷新机制
- 遇到 401 时，如果存在 refresh token，会尝试刷新后重放请求
- 如果刷新失败，则引导重新登录

## 错误处理约定
- 网络异常：统一提示网络连接异常
- 请求超时：统一提示接口请求超时
- HTTP 状态异常：统一提示系统接口异常
- 业务异常：优先使用后端 `msg`

## GET 参数处理
- 前端对 GET 请求参数会做对象展开和 URL 拼接
- 复杂查询对象会以 `field[key]=value` 的形式展开

## 文件下载/导出
- 当前前端支持 `blob` 响应判断
- 如果导出接口实际返回的是错误 JSON，小文件场景下前端会尝试解析并按错误处理

## 建议约束
- 新接口统一返回 `code/msg/data`
- 分页接口补充统一分页字段约定
- 导出接口明确 `blob` 返回与错误 JSON 返回规范
- 涉及长耗时接口，建议补充明确的超时、异步化或进度反馈策略
