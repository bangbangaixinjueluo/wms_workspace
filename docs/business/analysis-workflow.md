# 遗留系统业务文档补全工作流

## 1. 目标与适用范围

本工作流用于：
1. 逆向整理遗留系统业务规则
2. 在缺少文档的情况下，从代码中重建模块职责、状态流和数据关系
3. 形成可供后续 AI 持续接力、分析和改造的文档体系

适用场景：
1. Spring Boot 等已有代码但缺乏业务文档的系统
2. 需要按模块逐步补全文档，而不是一次性全量梳理
3. 需要支持后续 AI 在新会话中继续分析、回填或改造

---

## 2. 证据原则与标记体系

### 2.1 证据原则

分析代码时，只采信以下来源：

| 采信来源 | 说明 |
|----------|------|
| 条件判断（if/else/switch/三元） | 业务规则的直接体现 |
| 数据库操作（查询条件、写入字段、更新条件） | 数据事实 |
| Redis 操作 | 缓存/锁语义 |
| MQ 收发 | 异步副作用 |
| 状态赋值（`status = X`） | 状态机转换 |
| 异常抛出条件 | 前置约束 |
| `@Transactional` 注解 | 事务边界 |
| `@Scheduled` / `@Async` / `@EventListener` | 隐式入口和异步边界 |
| AOP / 拦截器 / 过滤器 | 隐式业务逻辑 |

一律不采信：
1. 方法注释
2. 字段注释
3. 行内注释
4. 类头部注释

注释与代码矛盾时，不采信注释，在文档中标记：

> ⚠️ 逻辑可疑：注释描述为“XXX”，但实际代码执行的是“YYY”，需确认以哪个为准

### 2.2 标记体系

| 标记 | 含义 |
|------|------|
| `⚠️ 逻辑可疑` | 代码逻辑看起来有问题，需人工确认 |
| `❓ 含义不明` | 字段/变量含义无法从代码推断，需人工确认 |
| `🔗 待回填` | 跨模块调用，被调用方尚未分析 |
| `🌐 外部系统` | RPC/HTTP，结果不在本地控制范围内 |

### 2.3 标记记录规则

1. 标记可以出现在模块正文对应位置，方便阅读上下文。
2. 所有标记必须同步登记到集中表格，不能只散落在正文中。
3. 集中表格作为唯一的待处理清单，用于阶段收尾、模块关闭和跨会话检索。

### 2.4 `🔗 待回填` 使用规则

1. 仅当当前文档已出现跨模块调用，但被调用方法的实际实现尚未阅读时，才允许标记 `🔗 待回填`。
2. 一旦已经读到被调用方法实现，并能从条件判断、状态赋值、数据库操作中得出结论，必须立即把结论回填到原文档并清除 `🔗`。
3. 不允许把“已经分析过但还没手工改文档”的内容继续保留为 `🔗 待回填`。

## 3. 标准执行流程

### 3.1 第零阶段：模块盘点

适用场景：
1. 只能先提供完整模块清单，暂时无法一次性提供全量接口签名或全量源码

输入：
1. 全部模块清单

执行内容：
1. 基于模块清单建立系统模块地图
2. 识别模块命名是否存在重叠、包含关系或潜在合并项
3. 先生成 `overview.md` 的模块列表骨架
4. 先定义分析批次和预期回填方向：
   - 当前模块依赖已分析模块：分析时必须即时回填
   - 当前模块依赖未分析模块：暂记 `🔗 待回填`
   - 当前模块依赖清单外模块：标记 `⚠️ 模块缺失/边界未明`
5. 产出建议分析顺序，等待人工确认

输出：
1. 模块列表
2. 模块分析顺序
3. `overview.md` 初版骨架

说明：
1. 在只有模块清单的前提下，不要求此阶段得出准确依赖关系。
2. 此阶段目标是确定分析顺序和回填闭环范围，不是输出业务结论。

### 3.2 第一阶段：全局扫描

输入分两批提供。

批次 1：
1. 项目目录结构（`tree` 命令输出）
2. `pom.xml` / `build.gradle`
3. 所有业务接口的完整方法签名
4. 数据库表结构 DDL

批次 2：
1. 所有 `@Aspect` / `Filter` / `HandlerInterceptor` 类源码
2. 所有 `@Scheduled` / `@Async` / `@EventListener` 类源码

执行内容：
1. 识别模块划分和技术栈
2. 从方法签名初筛模块列表和入口类型，入口类型以注解为准，签名只作初筛
3. 从表结构提取数据模型、字段含义、表间关联
4. 读取批次 2 源码，扫描隐式入口
5. 检测循环依赖，有环则合并为高内聚簇联合分析
6. 对模块做拓扑排序，确定分析顺序
7. 生成 `overview.md` 骨架和各模块文档框架

输出：
1. 模块列表和分析顺序
2. `overview.md` 骨架
3. 各模块文档框架

说明：
1. 若第零阶段已确认模块列表和分析顺序，此阶段重点补入口、表结构和初始依赖。

### 3.3 第二阶段：逐模块分析

每个模块按依赖顺序处理，每次只传当前模块的源码。

输入要求：
1. 至少保证当前模块源码和当前模块直接依赖的已分析模块结论可用
2. 若当前模块调用到清单内但尚未分析的模块，允许暂留 `🔗 待回填`
3. 若当前模块调用到不在模块清单中的模块，必须标记 `⚠️ 模块缺失/边界未明`

每个模块的分析步骤：

1. 读 Entity/DTO 源码
   - 只从字段赋值位置、查询条件、判断条件推断含义
   - 结合第一阶段表结构 DDL 交叉验证
   - 无法确定含义的字段标记 `❓ 含义不明`
2. 沿方法调用链追踪到实际执行点
   - 包含策略类、监听器、Mapper XML、自定义 SQL
   - 逐方法梳理执行步骤并附行号
   - 记录条件分支的每条路径
   - 记录数据库操作、Redis 操作、MQ 操作
   - 记录事务边界和 `rollbackFor`
   - 记录并发控制（幂等键、唯一约束、乐观锁、分布式锁）
   - 跨模块调用：已分析则直接引用，未分析则标记 `🔗 待回填`
   - 若分析过程中已读到被调用实现，必须同步回填原模块文档中的调用结论，并清除对应 `🔗`
   - 外部 RPC/HTTP 调用标记 `🌐 外部系统`
   - 逻辑可疑处标记 `⚠️ 逻辑可疑`
3. 将分析结果填入模块文档框架
4. 输出模块文档草稿
   - 头部附 commit hash 和分析日期
   - 同步更新 `overview.md` 中的全局标记表
   - 每新增一个 `⚠️ / ❓ / 🔗 / 🌐`，都要登记来源模块、方法、简述、状态、目标模块
5. 人工审阅草稿
   - 逐一确认 `⚠️ / ❓ / 🔗`
   - 确认结果写入文档并清除标记

可选补充：
1. 有数据库访问权限时，可对含义不明的枚举/状态字段做数据探查验证

### 3.4 第三阶段：收尾

1. 检查所有 `🔗 待回填` 是否已全部填完
2. 补全 `overview.md`
   - 模块调用关系
   - 共享数据表读写方
   - 隐式入口汇总
3. 人工最终审阅 `overview.md`

## 4. 模块完成与回填规则

### 4.1 模块完成后的强制回填

每完成一个模块，除更新该模块正文外，必须同步执行以下动作：

1. 检索所有已分析模块文档中 `目标模块 = 当前模块` 的记录。
2. 检索当前模块文档中的“集中待办和回填”表。
3. 检索 `overview.md` 中“全局集中待办和回填 / 标记跟踪表”里：
   - `来源模块 = 当前模块`
   - `目标模块 = 当前模块`
4. 凡是当前模块分析已经足以给出结论的，必须立刻回填正文。
5. 回填正文后，必须同步更新：
   - 当前模块文档的“集中待办和回填”表
   - `overview.md` 的全局表

### 4.2 回填判定规则

1. 已知调用关系，但未读到被调用实现：保留 `🔗 待回填`
2. 已读到被调用实现，且已确认状态流、条件、写库行为：立即回填，不得继续保留 `🔗`
3. 已读到被调用实现，但结论仍依赖更下游模块：只对仍未知的那一层继续保留 `🔗`

### 4.3 回填后的清理规则

1. 不允许出现“正文已写明结论，但模块集中表仍显示待回填”。
2. 不允许出现“模块集中表已改完，但 `overview.md` 全局表未同步”。
3. 不允许出现“其实已经读过实现，但因为还没改文档，继续保留 `🔗 待回填`”。
4. 原来的 `🔗` 在回填后，应当：
   - 直接删除；或
   - 改成回填说明；或
   - 融入正式业务逻辑描述

### 4.4 公共规则沉淀

当某个模块中出现被多个模块复用的公共规则时，不能只在调用方零散回填，必须在该模块文档中单列章节沉淀。

常见类型：
1. 公共状态机
2. 公共校验规则
3. 公共回写规则
4. 公共批处理规则

要求：
1. 公共规则应写入所属模块文档中的独立章节
2. 后续其他模块若依赖该规则，优先引用该章节，不重复新增等价 `🔗`
3. 若历史文档中已有等价 `🔗`，在公共规则沉淀后，应优先清理并改为正式结论或引用说明

### 4.5 模块完成判定

一个模块只有在以下条件同时满足时，才可视为本轮完成：

1. 模块正文已完成
2. 指向已分析模块的 `🔗` 已全部清理
3. 仅允许保留目标模块尚未分析的 `🔗`
4. 当前模块文档的“集中待办和回填”表已完成一次全量检索和状态刷新
5. `overview.md` 中与当前模块相关的全局记录已完成一次全量检索和状态刷新
6. 若本模块产出了公共规则，则历史文档中的等价待回填项已优先处理
7. 所有 `⚠️` 和 `❓` 已经人工确认，或明确标记为待业务确认的正式遗留项

## 5. 文档产物清单

### 5.1 基础产物

必须维护以下基础文档：
1. `docs/business/analysis-workflow.md`
2. `docs/business/overview.md`
3. `docs/business/{模块}.md`
4. `docs/business/session-handoff.md`

### 5.2 面向后续 AI 改造的必需产物

仅有模块业务文档还不够。若目标是支持后续使用 AI 安全地新增或修改系统，则必须额外维护以下产物：

1. 实体关系图
2. 状态机图
3. 跨模块回写清单
4. 外部系统交互清单
5. 字段语义表
6. 变更影响清单

这些产物与模块文档同级，不是可选补充项。

### 5.3 产物定义

#### 5.3.1 实体关系图

目标：让后续修改者快速理解“谁拥有数据、谁依赖数据、谁改谁”。

要求：
1. 使用 Mermaid 绘制，优先放在 `docs/business/model/` 目录
2. 至少维护两层：
   - 全局核心实体图
   - 核心业务链局部图
3. 关系只采信外键字段、查询条件、回写字段、聚合读取与批量更新
4. 不依据注释猜测实体关系

建议文件：
1. `entity-overview.md`
2. `entity-customer-order-chain.md`
3. `entity-transfer-chain.md`

#### 5.3.2 状态机图

目标：明确状态、迁移方向、触发动作和前置条件。

要求：
1. 对核心实体单独维护状态机图
2. 每条状态迁移必须写明：
   - 触发方法
   - 前置条件
   - 状态赋值证据
3. 至少覆盖：
   - 客户订单
   - 客留存
   - 产品包
   - 调拨单 / 成品调拨单
   - 出库结算单
   - 出货单

#### 5.3.3 跨模块回写清单

目标：避免后续改动时只改当前模块，漏掉共享表和下游联动。

要求：
1. 集中记录“来源模块 -> 目标模块/目标表 -> 写入字段 -> 触发动作 -> 证据位置”
2. 只记录真实写操作，不记录纯查询依赖
3. 模块正文中的跨模块写操作，必须同步登记到集中清单

#### 5.3.4 外部系统交互清单

目标：区分“本地业务完成”和“外部请求已发起/已回调成功”。

要求：
1. 对每个 RPC / HTTP / MQ / 回调交互单独记录：
   - 外部系统名称
   - 触发入口
   - 请求时机
   - 成功回写
   - 失败回写
   - 是否异步回调完成
   - 幂等键 / 流水号
2. 不允许把“请求发起成功”写成“业务最终成功”

#### 5.3.5 字段语义表

目标：降低 AI 误判关键字段含义的风险。

要求：
1. 不追求全字段覆盖，优先覆盖关键字段：
   - 状态字段
   - 类型字段
   - 业务编号字段
   - 金额 / 金重 / 件数字段
   - 同步状态 / 调拨状态 / 外部状态字段
2. 每个字段必须带证据位置：
   - 赋值点
   - 判断点
   - 查询条件
3. 无法确认语义时，必须标 `❓ 含义不明`

#### 5.3.6 变更影响清单

目标：让后续 AI 在改一个实体或字段前，知道必须同步检查哪些模块。

要求：
1. 以核心表或核心字段为维度维护影响面
2. 每项至少包含：
   - 变更对象
   - 受影响模块
   - 影响原因
   - 证据位置
3. 优先覆盖：
   - 共享主表
   - 核心状态字段
   - 跨模块回写字段

### 5.4 产物维护规则

1. 每完成一个核心模块，必须同步补该模块涉及的实体、状态、关键字段
2. 每完成一条核心业务链，必须同步更新对应实体关系图和状态机图
3. 每发现一个新的跨模块写操作，必须同步更新跨模块回写清单
4. 每发现一个新的外部系统调用或回调链路，必须同步更新外部系统交互清单
5. 每发现一个后续修改高风险字段，必须同步更新字段语义表和变更影响清单

## 6. 跨会话接力规范

若希望在新的 AI 会话中继续当前任务，必须把工作流设计成“会话无关、文档驱动”的形式，而不是依赖历史对话上下文。

### 6.1 必需交接文件

必须维护：
1. `docs/business/session-handoff.md`

该文件是新会话启动时的最小上下文入口，必须始终可用、可读、可直接继续执行。

### 6.2 `session-handoff.md` 必填内容

每次结束当前工作前，必须刷新以下内容：
1. 当前基准 commit 和日期
2. 当前分析范围
3. 已完成模块
4. 进行中模块
5. 下一优先模块
6. 当前未关闭的 `⚠️ / ❓ / 🔗 / 🌐`
7. 本轮已确认的关键结论
8. 下一会话建议读取的文档顺序

### 6.3 新会话启动顺序

在新会话中继续任务时，必须按以下顺序读取：
1. `docs/business/analysis-workflow.md`
2. `docs/business/session-handoff.md`
3. `docs/business/overview.md`
4. 当前目标模块文档
5. 相关实体关系图
6. 相关状态机图
7. 相关总表类产物：
   - 跨模块回写清单
   - 外部系统交互清单
   - 字段语义表
   - 变更影响清单

### 6.4 结论落库规则

所有已确认结论必须写入仓库文档，不允许只停留在对话中。

至少包括：
1. 新确认的状态流
2. 新确认的字段语义
3. 新确认的跨模块回写
4. 新确认的外部系统交互
5. 已关闭的 `⚠️ / ❓ / 🔗 / 🌐`

若某条结论未写入文档，则视为未完成，不能假设下一会话会继承该信息。

### 6.5 会话结束前检查

每次准备结束当前会话前，必须至少完成以下检查：
1. 当前模块正文是否已更新
2. 模块“集中待办和回填”是否已更新
3. `overview.md` 是否已同步
4. 对应图表或总表是否已同步
5. `session-handoff.md` 是否已刷新

只有完成以上同步后，当前会话才能视为可安全交接。

---

## 7. AI 使用前置检查

后续若使用 AI 对系统做新增或修改，分析上下文至少必须包含以下资料：

1. 当前模块文档
2. `overview.md`
3. 对应业务链的实体关系图
4. 对应核心实体状态机图
5. 对应字段语义表
6. 对应变更影响清单
7. 若涉及对外联动，必须额外包含外部系统交互清单

不允许仅凭接口文档或单模块文档直接生成改造方案。

## 8. 文档结构与模板

### 8.1 文档结构

```text
docs/
└── business/
    ├── analysis-workflow.md
    ├── overview.md
    ├── session-handoff.md
    ├── cross-module-writeback.md
    ├── external-interactions.md
    ├── field-semantics.md
    ├── change-impact.md
    ├── {模块A}.md
    ├── {模块B}.md
    └── model/
        ├── entity-overview.md
        ├── entity-xxx.md
        ├── xxx-state.md
        └── ...
```

### 8.2 `overview.md` 模板

### 8.3 `{模块}.md` 模板

```markdown
# {模块名}
> 基于 commit: `{hash}`，日期：{date}

## 模块职责
一句话说明这个模块管什么数据、负责什么操作。

## 涉及数据表
| 表名 | 读/写 | 说明 |
|------|-------|------|
| xx_order | 读写 | 主表 |
| xx_item  | 读写 | 明细表 |

## 接口列表

### {方法名}
> `XxxServiceImpl.java#methodName()` L{行号}

- **入口类型**：HTTP 接口 / 定时任务 / MQ 消费 / 事件监听 / 内部调用
- **用途**：
- **入参**：
  - `字段名` - 类型 - 含义
- **出参**：
  - `字段名` - 类型 - 含义
- **事务边界**：`@Transactional` 范围，`rollbackFor` 条件 `→ L{行号}`
- **并发控制**：幂等键 / 唯一约束 / 分布式锁 / 乐观锁字段
- **业务逻辑**：
  1. 步骤描述 `→ L{行号}`
  2. 调用 `XX模块.XX方法()` `→ L{行号}`
  3. 🌐 调用外部系统 XX，影响：...
- **外部依赖**：
  - MQ：发送 topic `xxx`，时机：...
  - Redis：key `xxx`，操作：读/写/删/加锁
- **状态变更**：表名.字段 从 X → Y，条件：... `→ L{行号}`
- **异常情况**：条件 → 抛出 XxxException `→ L{行号}`
- **⚠️ 逻辑可疑**：（无则省略）
- **❓ 含义不明**：（无则省略）
```
