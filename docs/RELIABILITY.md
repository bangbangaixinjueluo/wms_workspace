# RELIABILITY

## 当前可靠性基线
结合前后端代码，项目已经具备一定的可靠性基础，但整体仍偏“经验型保障”，自动化验证和环境隔离还有提升空间。

## 已具备的能力
### 前端
- 请求层统一封装，支持超时、错误提示和 token 刷新。
- 登录失效、重复登录、后端异常码有集中处理。
- 路由守卫能在未取到权限数据时阻断页面访问。

### 后端
- Redis 作为缓存与部分分布式能力基础设施。
- Quartz 支持定时任务。
- Flowable 支持流程状态留痕。
- Actuator、Spring Boot Admin、Druid、日志配置可用于运行观测。
- Lock4j、Resilience4j 等依赖已接入，具备做限流与锁控制的基础。

## 当前主要风险
### 构建回归风险
- 根 `pom.xml` 默认跳过测试，意味着后端构建不执行单测。
- 前端仓库中未看到成体系的自动化测试目录，页面回归主要依赖人工验证。

### 配置漂移风险
- 前后端均存在较强的环境内嵌配置。
- 一旦多人并行调试或切换环境，容易出现“本地可用、他处不可用”的问题。

### 首次加载压力
- 前端登录后会一次性加载较多基础数据。
- 当接口响应慢、字典量大或网络不稳定时，首屏可用时间会被拉长。

### 单体扩展压力
- 后端是模块化单体，业务域持续增长后，单进程启动时间、回归范围和排障复杂度都会上升。

## 建议的可靠性改进
### 构建链路
- 开启后端最小单测集，不建议长期全局 `skip=true`。
- 为前端至少补充登录、动态路由、核心表格页的冒烟验证。

### 配置治理
- 明确 `local/dev/test/uat/prod` 的配置来源，避免把环境特定信息直接固化在仓库。
- 将数据库、Redis、第三方凭据迁移出源码仓库。

### 运行观测
- 固化日志字段，至少覆盖用户、租户、单据号、仓库、接口耗时、异常码。
- 对关键任务增加运行结果记录与失败告警。

### 业务保障
- 涉及库存和财务的一致性变更，优先定义幂等边界、补偿方式和人工修复流程。
- 外部系统同步任务应区分实时链路与补偿链路，避免同步失败直接阻塞主流程。

## 核心验证建议
- 登录、刷新 token、退出登录
- 动态菜单和按钮权限
- 仓储核心单据创建、审核、反审核
- 报表查询与导出
- 定时任务与外部同步链路
