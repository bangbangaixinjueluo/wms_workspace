# PRODUCT_SENSE

## 产品定位判断
从 `wms-web` 与 `wms-api` 的目录、模块和脚本可以判断，当前系统不是单一仓库台账，而是一个面向企业内部运营的综合业务平台：

- 底座是通用管理后台能力
- 核心价值在仓储、商品、订单、财务、盘点、调拨、报表
- 同时承载审批流、系统管理、商城与第三方业务同步

更贴切的描述是：`企业定制化 WMS/业务协同平台`。

## 主要用户类型
根据页面和模块划分，可以推测当前主要用户包括：

- 系统管理员：维护用户、角色、菜单、配置、任务、字典。
- 仓库/库位人员：处理入库、出库、盘点、调拨、库存查询。
- 业务/销售人员：处理订单、销售、客户、留存和报表。
- 财务或结算人员：处理财务、账户、应收应付相关流程。
- 审批人员：处理 BPM 流程与业务单据审批。

## 核心业务对象
从 `src/views/wh` 和 `src/api/wh` 的目录可以看到，产品的核心对象主要是：

- 商品 / SKU / 款式 / 材料 / 配件
- 仓库 / 库位 / 库存
- 客户 / 供应商 / 商户
- 订单 / 销售单 / 调拨单 / 盘点单
- 财务与结算对象
- 统计报表对象

## 主流程判断
### 运营主流程
- 用户登录系统
- 进入对应业务域页面
- 查询并筛选基础数据
- 处理单据、库存、订单或财务动作
- 通过按钮权限控制不同角色的操作范围
- 在报表与统计页面查看经营结果

### 平台主流程
- 管理员维护菜单、角色、字典、定时任务与配置
- 若业务接入审批，则由 BPM 负责流转和留痕
- 若业务接入外部系统，则由后端同步模块负责数据交换

## 产品特征
- 权限驱动明显：页面、菜单、按钮都强依赖权限码。
- 字典驱动明显：状态、类型、展示文本大量依赖字典。
- 单据驱动明显：从旧 README 可见项目内部大量业务围绕“单据状态流转”展开。
- 报表诉求较强：目录中存在多个报表页面、SQL 报表脚本和报表模块。

## 需求分析时的判断原则
- 优先区分“基础平台诉求”和“仓储业务诉求”，避免把平台问题误当业务规则。
- 所有单据类需求都要先确认状态流转、审批规则和可逆操作。
- 涉及库存、财务、同步的变更，都默认视为高风险需求，需要先考虑回滚与补偿。
