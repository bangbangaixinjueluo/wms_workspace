# PLANS

## 目的
本文用于约束本仓库在 `docs/exec-plans` 下维护执行计划的方式，确保问题处理、重构、专项优化都能留下可追踪记录。

## 目录约定
- `docs/exec-plans/active/`：进行中的计划
- `docs/exec-plans/completed/`：已完成计划
- `docs/exec-plans/tech-debt-tracker.md`：长期技术债和跨计划问题

## 适合建立执行计划的场景
- 跨前后端联动改造
- 涉及数据库脚本或数据迁移的任务
- 仓储、财务、审批等关键链路优化
- 大于 1 天工作量的重构或专项治理
- 需要分阶段验收的文档补齐工作

## 推荐模板
```md
# 计划标题

## 背景

## 目标

## 范围

## 非目标

## 实施步骤
1. ...
2. ...
3. ...

## 风险

## 验证方式

## 当前状态
- draft / active / blocked / completed / cancelled
```

## 本仓库特别建议
- 涉及 `wms-web` 和 `wms-api` 同时变更时，计划中必须拆开前端、后端、联调三段。
- 涉及 `sql/update` 或 `sql/version` 时，计划中必须写清楚回滚思路。
- 涉及权限模型、审批流、库存或财务口径时，必须写清业务确认前提。

## 当前建议优先立项的计划
- 敏感配置治理与环境解耦
- 前端登录后基础数据预加载梳理
- 后端模块说明与数据库变更规范补齐
- WMS 核心单据状态流转文档化
