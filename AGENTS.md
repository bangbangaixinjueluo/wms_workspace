# AGENTS

## 本文用途
用于定义人类开发者与 AI Agent 在本仓库内的协作方式、修改边界、交付标准与沟通约定。任何自动化执行、批量修改、文档生成或代码重构，都应先遵守本文。

## 适用范围
适用于整个工作空间，包括源码、脚本、配置、文档、数据定义与生成产物目录。

## 如何维护
- 协作规则发生变化时优先更新本文。
- 新增高风险自动化能力前，先补充边界与审计要求。
- 若本文与更细粒度模块规范冲突，以模块规范为准，并在本文记录例外。

## 协作约定
### 角色分工
- 人类开发者负责产品目标、业务判断、上线决策与最终验收。
- Agent 负责信息收集、方案实现、文档维护、测试执行与风险提示。
- 对于高风险变更，Agent 需要明确说明影响范围与回滚思路。

### 沟通规则
- 所有变更默认采用最小必要修改原则。
- 无法确认的业务语义，不擅自推断为最终规则。
- 对现有非本人改动保持兼容，不主动覆盖。

## 修改原则
### 默认策略
- 优先修正根因，不做表面兜底。
- 优先保持现有接口兼容，除非已有明确升级计划。
- 避免一次性引入与目标无关的重构。

### 文件处理
- 生成文件与手写文件应明确区分。
- 未经确认，不修改密钥、凭据、部署参数与生产数据脚本。
- 文档更新应与代码行为保持一致。

## 执行边界
### 可直接执行
- 非破坏性的代码修复与文档初始化。
- 新增测试、补充注释、完善说明文档。
- 对局部问题做小范围重构。

### 需要额外确认
- 数据迁移与批量删除。
- 对外 API 破坏性变更。
- 权限模型、计费逻辑、风控规则的调整。

## 交付要求
- 说明做了什么、为什么这样做、还有什么未覆盖。
- 对关键路径提供验证方式。
- 对暂未完成项给出清晰的待补充标记。
